# 5.1 Gestión de Estado en Aplicaciones Web

El protocolo HTTP es, por diseño, un protocolo *stateless*: cada petición se
procesa de forma independiente y el servidor no conserva información sobre las
peticiones anteriores del mismo cliente. Esta característica favorece la
escalabilidad horizontal, porque cualquier instancia del servidor puede atender
cualquier petición sin depender de un estado local. Sin embargo, las aplicaciones
reales requieren mantener estado —sesiones de usuario, autenticación,
preferencias—, lo que obliga a introducir mecanismos externos que reconcilien la
naturaleza stateless de HTTP con la necesidad de continuidad (Bucko et al.,
2023).

## Mecanismos de almacenamiento del lado del cliente

Existen tres mecanismos principales para almacenar estado en el navegador, cada
uno con características distintas:

| Característica | localStorage | sessionStorage | Cookies |
|---|---|---|---|
| Capacidad | ~5-10 MB | ~5-10 MB | ~4 KB |
| Persistencia | Permanente | Hasta cerrar pestaña | Configurable (Max-Age) |
| Accesible desde JS | Sí | Sí | Solo si no es HttpOnly |
| Envío automático al servidor | No | No | Sí |
| Riesgo XSS | Alto | Alto | Bajo (con HttpOnly) |
| Riesgo CSRF | Bajo | Bajo | Alto (mitigable con SameSite) |

La elección entre estos mecanismos depende de un compromiso de seguridad.
Almacenar un token de autenticación en `localStorage` lo hace legible por
cualquier script inyectado mediante un ataque de *cross-site scripting* (XSS),
ya que este almacenamiento es accesible a cualquier script que se ejecute en el
contexto de la página (OWASP Foundation, 2021). Por el contrario, una cookie con
el atributo `HttpOnly` es inaccesible desde JavaScript, lo que impide que un
script lea el token y lo exfiltre, mitigando así el robo por XSS; a cambio,
introduce exposición a *cross-site request forgery* (CSRF), mitigable mediante el
atributo `SameSite`.

En el presente sistema (SGED) se optó por almacenar el JWT en una cookie
HttpOnly con `SameSite=Lax`, priorizando la protección contra XSS. El token nunca
se expone en el cuerpo de la respuesta ni queda accesible al JavaScript del
cliente, en línea con las prácticas de autenticación stateless basada en JWT
sobre cookies HttpOnly documentadas en la literatura reciente (Bucko et al.,
2023).

## Estado de sesión en arquitecturas multi-instancia

En arquitecturas con múltiples instancias de servidor, la gestión de sesión
presenta un desafío adicional. El almacenamiento de sesión en memoria local
obliga a usar *sticky sessions*, que atan cada cliente a una instancia específica
y limitan la escalabilidad y la tolerancia a fallos: si esa instancia cae, se
pierde la sesión del usuario. La solución adoptada consiste en externalizar el
estado —en este caso, la blacklist de tokens revocados— a un almacén centralizado
en memoria como Redis, de modo que todas las instancias comparten la misma vista
del estado sin acoplarse entre sí. Redis ofrece acceso en memoria de baja
latencia con expiración automática por TTL, lo que lo hace idóneo para datos de
sesión de vida corta y para reducir la carga sobre la base de datos relacional
(Privalov & Stupina, 2024).

## Ciclo Request/Response en Spring Boot

En el marco de Spring Boot, el ciclo *request/response* se procesa mediante una
cadena de filtros de servlet. Para la autenticación stateless con JWT, un filtro
personalizado (`JwtAuthenticationFilter`) intercepta cada petición, extrae el
token de la cookie HttpOnly, valida su firma y expiración, y **consulta la
blacklist en Redis** por el identificador único del token (JTI) antes de
establecer el contexto de seguridad. Este paso garantiza que un token revocado
mediante logout no pueda autenticar aunque siga siendo criptográficamente válido
y no haya expirado, resolviendo la principal limitación de los JWT stateless: la
dificultad para revocarlos antes de su expiración natural (Bucko et al., 2023).
